## 🌟 Overview

The **YOLO Dataset Validation and Benchmarking Service** is a specialized evaluation component within the `agridrone-vision-evaluation-pipeline` project.

Its purpose is to execute reproducible validation of trained YOLO models over agricultural datasets while controlling the execution environment, dataset split selection, GPU warm-up, metric extraction, timing measurement, and experiment tracking.

This component is more specific than the general CLI orchestrator. The CLI decides *when* validation should run, while this service defines *how* validation is executed, measured, logged, and persisted.

Typical trigger:

```bash
python -m scripts.main
```

The user selects the validation workflow, dataset/project, YOLO version, model size, image size, batch size, confidence threshold, and GPU/CPU device. The CLI then invokes the validation module.

## 🎯 Main Responsibility

The service executes reproducible YOLO validation over agricultural datasets and persists results for analysis, comparison, and traceability.

It is responsible for:

- resolving trained model paths
- loading `best.pt`
- reading training metadata such as `args.yaml`
- extracting run seed when available
- generating temporary validation YAML files
- redirecting validation toward the intended dataset split
- selecting GPU/CPU execution device
- warming up the model on GPU
- clearing CUDA cache between runs
- executing `model.val()`
- extracting global and per-class metrics
- measuring runtime and timing statistics
- logging metrics into ClearML
- saving structured JSON summaries
- supporting reproducible benchmarking across multiple runs

## 🔍 Detailed System Flow

### Step 1: CLI Trigger

The workflow starts from the main CLI:

```bash
python -m scripts.main
```

The user selects:

- execution mode
- dataset/project
- YOLO version
- model size
- `img_size`
- batch size
- confidence threshold
- GPU or CPU device

The CLI then routes execution to the validation service.

---

### Step 2: Resolve Model and Metadata

The service resolves the trained model path and supporting metadata.

Expected model artifact:

```text
best.pt
```

Expected metadata artifact:

```text
args.yaml
```

The `args.yaml` file may be used to recover training configuration and seed information.

Technical concern:

```text
The component is implicitly coupled to the directory structure generated by Ultralytics.
```

---

### Step 3: Extract Seed from `args.yaml`

When available, the validation service extracts the seed used during training.

Purpose:

- improve reproducibility
- connect validation results to the original training run
- support experiment traceability

If `args.yaml` is missing or incomplete, the service should log the issue and continue with explicit metadata indicating that seed recovery was unavailable.

---

### Step 4: Generate Temporary Validation YAML

The service may generate a temporary YAML file to redirect validation toward the desired split, commonly `test/images`.

Example:

```text
temp_val.yaml
```

Purpose:

- preserve the original dataset YAML
- validate a specific split without permanently modifying dataset configuration
- support controlled benchmarking against test or validation subsets

Technical risk:

```text
If path resolution is incorrect, temporary YAML generation can produce invalid paths such as test/images/test/images.
```

Recommended validation:

- verify that the generated YAML exists
- verify that image paths resolve correctly
- verify that `names` and `nc` match label files
- verify that the target split contains images
- verify write permissions before creating temporary files

---

### Step 5: GPU Warm-up

Before benchmarking, the service performs a GPU warm-up when CUDA is available.

Purpose:

- stabilize runtime measurements
- reduce first-call overhead
- avoid overestimating average inference time
- initialize CUDA kernels and model execution path

Failure handling:

```text
Warm-up failure should be logged but should not necessarily stop validation if the model can still run.
```

---

### Step 6: CUDA Cache Cleanup

The service clears CUDA cache between runs when applicable.

Example operation:

```python
torch.cuda.empty_cache()
```

Purpose:

- reduce memory contamination between validation runs
- avoid inflated memory usage
- improve repeatability when multiple models or runs are evaluated sequentially

Technical concern:

Cache cleanup does not guarantee full determinism. It should be treated as a memory hygiene step, not as a complete reproducibility mechanism.

---

### Step 7: Execute YOLO Validation

The core validation step uses Ultralytics:

```text
model.val()
```

Validation configuration may include:

```text
imgsz
batch
conf
split
device
project
name
```

The service evaluates the model against the configured dataset split and produces Ultralytics validation artifacts.

---

### Step 8: Extract Global Metrics

The service extracts global detection metrics from validation results, typically from `results.box`.

Core metrics:

```text
precision
recall
mAP50
mAP50-95
```

These metrics summarize model performance across classes.

---

### Step 9: Extract Per-Class Metrics

The service also extracts per-class metrics, commonly as arrays.

Technical risk:

```text
Array-based per-class metrics are fragile if class order changes or if class names are not explicitly attached to each metric value.
```

Recommended output structure:

```json
{
  "class_metrics": [
    {
      "class_id": 0,
      "class_name": "weed",
      "precision": "ILLUSTRATIVE_VALUE",
      "recall": "ILLUSTRATIVE_VALUE",
      "map50": "ILLUSTRATIVE_VALUE",
      "map50_95": "ILLUSTRATIVE_VALUE"
    }
  ]
}
```

This is safer than persisting unnamed arrays.

---

### Step 10: Compute Timing Statistics

The service computes performance statistics across runs.

Useful timing fields:

```text
average_time_per_image
inference_time
postprocess_time
preprocess_time
std_time_between_runs
images_per_second
```

Why this matters:

- supports model/runtime comparison
- helps evaluate GPU efficiency
- informs deployment planning
- quantifies cost of larger image sizes or batch sizes

---

### Step 11: Log Metrics to ClearML

When ClearML is enabled, the service logs:

- global metrics
- per-class metrics
- timing statistics
- validation configuration
- artifacts
- run metadata

Scope clarification:

```text
ClearML is an external tracking integration. Validation should not fail completely only because ClearML logging fails.
```

Recommended approach:

- isolate ClearML failures from core validation
- log ClearML errors separately
- continue local JSON persistence even if remote tracking fails

## 🚨 Technical Risks

### Tight Coupling to Ultralytics Folder Structure

The service depends on artifacts such as:

```text
args.yaml
results.csv
weights/best.pt
```

Risk:

- changes in Ultralytics folder conventions may break path resolution
- copied or moved training folders may lose required metadata

Mitigation:

- validate expected files before execution
- store resolved paths in a run manifest
- avoid relying only on implicit folder traversal

---

### Temporary YAML Path Bugs

Dynamic YAML modification can introduce path bugs.

Known example:

```text
test/images/test/images
```

Mitigation:

- normalize paths before writing YAML
- validate generated YAML immediately
- use absolute resolved paths only after checking root paths
- add unit tests for YAML generation

---

### Metrics by Class Stored as Arrays

Array-based class metrics can become ambiguous.

Risk:

- class ordering changes
- class names do not align with values
- downstream reports mislabel metrics

Mitigation:

- persist class metrics as named objects
- include `class_id` and `class_name`
- validate array length against `names` and `nc`

---

### ClearML Failure Coupling

If ClearML logging fails, core validation should still complete.

Mitigation:

- wrap ClearML logging separately
- persist local JSON first
- log ClearML status in the summary
- avoid making remote tracking a hard dependency

---

### Filesystem and Permission Issues

Mounted datasets and output directories may have permission issues.

Mitigation:

- validate read/write permissions before validation
- avoid writing cache files to read-only dataset roots
- allow configurable cache/output locations

---

## ⚠️ Known Failure Modes

| Failure Mode | Cause | Recommended Fix |
|---|---|---|
| YAML points to wrong split | Dynamic split rewriting bug | Validate temporary YAML before `model.val()` |
| `test/images/test/images` path | Path concatenation error | Normalize paths and avoid double-joining |
| Missing `args.yaml` | Training folder incomplete or moved | Continue with seed unavailable and log warning |
| GPU warm-up failure | CUDA/model initialization issue | Log warning and continue if validation can run |
| ClearML logging error | API or task configuration issue | Isolate tracking failure from validation |
| Dataset write permission error | Mounted dataset path read-only | Use separate writable cache/output directory |
| Class metric misalignment | Metrics stored as arrays | Persist metrics by `class_id` and `class_name` |
| Reused `temp_val.yaml` | Non-idempotent temporary file naming | Generate per-run temp YAML using `run_id` |

## 🧪 Recommended Validations Before Running

Before executing validation, check:

- `best.pt` exists
- `args.yaml` exists or is explicitly optional
- original `dataset.yaml` is valid YAML
- temporary YAML points to existing image directory
- labels directory exists
- `names` and `nc` are consistent
- class IDs in labels are valid
- GPU is available when selected
- output directory is writable
- ClearML is configured when enabled

## 🧠 Additional Implementation Contracts

### Metric Source Contract

Every validation summary should explicitly state where metrics came from:

```text
model.val()
post_training_validation
results.csv
ClearML artifact
```

This prevents ambiguity when training metrics are recovered after a run rather than returned directly by `model.train()`.

### Checkpoint Ownership Contract

Validation should not only receive a `best.pt` path; it should verify and persist checkpoint ownership.

Recommended fields:

```text
training_run_id
checkpoint_path
checkpoint_hash_or_file_size
source_args_yaml
source_results_csv
selected_metric_row
```

### `max_det` Recording

YOLO validation and inference may use `max_det`, which is distinct from COCO `maxDets`. The validation summary should persist `max_det` whenever it is configured.

### Resolution Compatibility Fields

Validation summaries should persist:

```text
train_img_size
validation_img_size
inference_img_size_if_applicable
sahi_slice_size_if_applicable
```

These fields are required to interpret model behavior across 2048/4K imagery and standard training resolutions.

### ClearML Local-First Policy

Local JSON persistence should complete even if ClearML logging fails. The validation summary should include:

```json
{
  "clearml_enabled": true,
  "clearml_status": "success | failed | disabled",
  "clearml_error": null
}
```

## 🚀 Production Hardening Recommendations

Priority improvements:

1. Replace reusable `temp_val.yaml` with per-run temp YAML files.
2. Validate temporary YAML before running `model.val()`.
3. Persist class metrics as named objects.
4. Isolate ClearML failures from core validation.
5. Add run-level manifest.
6. Add non-interactive CLI mode.
7. Add unit tests for YAML rewriting and path handling.
8. Add structured logging for each validation stage.
9. Add retry only for non-deterministic I/O and tracking failures.
10. Add configurable cache and output directories for mounted datasets.
